# INIT-25 — Risky Action Controls (Full Deep Revision)

## Scope
Issue: **INIT-25**
Objective: define concrete, enforceable controls for high-risk AI-agent actions across command execution, filesystem, network egress, and repository mutation domains.

This specification is meant to be operational (engineering), auditable (security/compliance), and explainable (customer credibility).

---

## 1) Why Risky Action Controls Need Their Own Layer
Detection alone does not prevent harm. For agentic tooling, the highest-impact failures occur when actions are allowed without contextual gating.

Risky action controls are the execution guardrails that sit between:
- intent generated by AI tools,
- and real-world side effects on systems, data, and codebases.

Design principle:
> "Permit useful automation, but require proportional friction as action impact increases."

---

## 2) Canonical Action Risk Taxonomy

### R1 — Low-impact read-only actions
Examples: non-sensitive file reads, benign metadata lookups.
Default: Detect/Warn depending on confidence and context.

### R2 — Scoped write/modify actions
Examples: edits in approved repo paths, non-privileged config changes.
Default: Warn or Approval Required in medium/high sensitivity contexts.

### R3 — Broad or boundary-crossing actions
Examples: high fan-out writes, external uploads, cross-repo mutations.
Default: Approval Required; Block when policy disallows.

### R4 — Privileged/destructive/prohibited actions
Examples: privilege escalation, credential store access, protected path writes, disallowed command classes.
Default: Block unless explicit high-trust, time-bound exception exists.

---

## 3) Control Domains

## A) Shell / Command Controls

### Core controls
- command allow/deny matrix by risk class,
- privilege-aware command gating (`sudo`, package/system-level mutation, service control),
- command-chain controls (sequence-aware checks, not just single command regex),
- high-risk command step-up approval.

### Enforcement examples
- deny known destructive patterns in restricted endpoints,
- require approval for system package install + execution chain,
- block chained commands attempting shell escape from constrained contexts.

### Evidence requirements
- command summary and normalized class,
- parent/child process lineage,
- privilege context,
- policy rule ID and decision trace.

---

## B) Filesystem Controls

### Core controls
- protected path denylist (secrets, key material, system configs, regulated data directories),
- path-scoped write allowlists by project/team,
- file fan-out thresholds triggering escalation,
- overwrite-risk checks for critical files.

### Enforcement examples
- allow edits only inside approved repo subtree,
- require approval for writes touching sensitive config paths,
- block writes to credential directories regardless of confidence tier.

### Evidence requirements
- target path set,
- write intensity metrics (files touched, overwrite ratio),
- sensitivity tags for each path,
- before/after hash references where feasible.

---

## C) Network / Egress Controls

### Core controls
- destination allowlist by environment and tool class,
- egress gating for sensitive sessions,
- upload/path-aware transfer restrictions,
- proxy-aware identity and attribution checks.

### Enforcement examples
- block outbound transfer to unsanctioned hosts,
- require approval for exports from regulated repositories,
- downgrade confidence or escalate when destination attribution is unresolved.

### Evidence requirements
- destination identity (hostname/SNI/IP),
- process-to-connection mapping,
- payload class (metadata only vs data-bearing transfer),
- policy decision and rule trace.

---

## D) Repository / SDLC Controls

### Core controls
- protected branch guardrails,
- mandatory PR/review gates for high-impact AI-assisted diffs,
- commit policy checks (identity/signature metadata where required),
- churn-based controls (LOC/files/components touched thresholds).

### Enforcement examples
- block direct writes to protected branches,
- step-up approval for broad critical-service refactors,
- require review linkage before merge-eligible actions.

### Evidence requirements
- repo/branch target,
- change-size metrics,
- review policy expectation vs observed behavior,
- commit lineage metadata.

---

## 4) Contextual Gating Logic
Risky action controls should evaluate action + context together.

Context inputs:
- confidence score,
- asset sensitivity,
- actor trust tier,
- endpoint posture,
- exception state,
- session history (prior warnings/denials).

Rule model:
- action class determines baseline control,
- context can only escalate or narrow within policy bounds,
- explicit deny policy always wins.

---

## 5) Approval and Exception Semantics

### Approval tokens must be:
- scope-limited (exact action/path/repo),
- time-limited,
- owner-bound,
- revocable.

### Exception requirements
- documented justification,
- compensating controls,
- expiration + reapproval policy,
- audit visibility.

### Anti-abuse protections
- no wildcard exceptions for Class C autonomous actions,
- repeated denied retries trigger escalation,
- stale exceptions auto-expire and revert baseline controls.

---

## 6) Decision Table (Representative)
1. R1 + low sensitivity + low confidence => Detect
2. R2 + medium sensitivity + medium confidence => Warn
3. R2 + high sensitivity => Approval Required
4. R3 + medium/high sensitivity => Approval Required
5. R3 + explicit destination/path deny => Block
6. R4 in any restricted environment => Block
7. Repeated warning bypass attempts => escalate to Approval/Block

Every row maps to deterministic rule IDs and explainability payloads.

---

## 7) Failure Modes and Mitigations

### Failure Mode: Overblocking productivity
Mitigation:
- confidence-aware gating,
- scoped approvals,
- targeted deny rules instead of blunt global blocks.

### Failure Mode: Underblocking high-impact automation
Mitigation:
- stricter defaults for R3/R4,
- protected-path hard denies,
- session-level escalation for repeat violations.

### Failure Mode: Policy drift and exception sprawl
Mitigation:
- periodic exception review,
- policy versioning and replay tests,
- ownership and expiry enforcement.

### Failure Mode: Poor audit defensibility
Mitigation:
- mandatory evidence schema,
- immutable decision traces,
- rule/version provenance on every enforcement event.

---

## 8) Data Model Requirements
Minimum risky-action event fields:
- event ID, timestamp,
- actor + host + session,
- action class + normalized action summary,
- targets (paths/repos/destinations),
- sensitivity tags,
- confidence score,
- decision state,
- rule ID/version,
- approval/exception linkage,
- evidence reference IDs.

---

## 9) Validation Plan

### Positive validation
1. allowed R1/R2 actions pass with expected low-friction behavior.
2. scoped approvals work and expire correctly.
3. repo guardrails trigger on protected branch scenarios.

### Adversarial validation
1. privilege escalation attempt (R4) blocked deterministically.
2. unsanctioned egress attempt blocked with explainable reason.
3. path-boundary bypass attempts escalated and denied.

### Evidence outputs required
- control efficacy report by action class,
- false-positive and false-block samples,
- approval lifecycle traces,
- residual risk statements.
