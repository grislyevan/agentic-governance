Detec Whitepaper

Endpoint Governance for Agentic AI Tools

See what AI agents do. Govern what they are allowed to.

Executive Summary

Agentic AI tools now execute shell commands, write files, open network connections, and run local inference directly on developer endpoints. Existing control stacks were not designed for this category. Cloud controls miss localhost-only operation, endpoint controls capture processes but struggle with attribution, and application inventories show installs without runtime behavior.

Detec is an endpoint agent that detects tools by capability rather than product name. It classifies behavior into four governance classes and applies confidence-scored, policy-driven enforcement from visibility to blocking.

Detec uses five signal dimensions with per-tool calibrated weights, computes an explainable confidence score, and evaluates deterministic policy rules that account for tool class, asset sensitivity, and action risk. Every decision includes rule ID, evidence contributors, and audit trail context.

1) The Governance Gap

Agentic tools are different because they can act autonomously, can run locally with little or no cloud visibility, can change risk profile at runtime, and can blend into legitimate developer infrastructure.

2) Detection Model

Detec scores five signal dimensions: Process, File, Network, Identity, and Behavior.

  • Process: binary identity, parent-child lineage, execution shape.
  • File: config footprints, caches, model storage, tracking databases.
  • Network: endpoints, listener ports, burst cadence, process-socket attribution.
  • Identity: account mapping, signatures, credential context, OAuth metadata.
  • Behavior: temporal execution patterns, prompt-edit-commit loops, fan-out writes, inference bursts.

A future sixth signal, binary hash verification (SHA-256), is planned for M3.

Classification is dynamic and based on evidence. When signals indicate multiple classes, the highest-risk applicable class is assigned with reason codes.

3) Confidence Scoring

Confidence is deterministic, auditable, and explainable.

base_score = sum(layer_weight * layer_signal_strength) penalty_total = sum(applicable penalties) evasion_boost = sum(evasion indicators) final = clamp(base_score - penalty_total + evasion_boost, 0.0, 1.0)

Per-tool weight vectors are calibrated to lab evidence. Penalties are applied for evidence gaps such as missing process lineage, unresolved process-network linkage, stale artifact-only conditions, or weak identity correlation.

If suppression behavior is detected (for example explicit attribution marker disablement), the model applies evasion boosts because suppression itself is governance-relevant evidence.

4) Policy Engine and Enforcement Ladder

The policy engine is deterministic and versioned. Inputs are confidence band, tool class, asset sensitivity tier (Tier 0–3), and action risk level (R1–R4). Rules produce one state: Detect, Warn, Approval Required, or Block.

Overlay rules for network correlation and isolation can escalate outcomes but do not downgrade them.

5) System Architecture

Detec operates as three tiers: endpoint collector, central API, and dashboard.

  • Collector: periodic scans, confidence and policy evaluation, structured events, retry logic, local queue fallback, heartbeat telemetry.
  • API: tenant-aware ingestion, JWT and API key auth, PostgreSQL storage, endpoint auto-registration.
  • Dashboard: fleet-level visibility into detections, confidence, policy state, and evidence context.

Event schema uses strict enums for decision-critical fields and is designed for direct SIEM ingestion.

6) Validation Methodology

Each detection profile is validated through repeatable lab runs on active tools. Indicators are marked confirmed, partial, or not observed, and weights are adjusted only when evidence warrants recalibration.

Evasion testing is a dedicated track. It documents suppression vectors, detection surfaces, and multi-signal counter-indicators. The objective is evidence-backed governance under realistic conditions, not perfection claims.

7) Known Limits

  • Containerized and remote development can reduce host-side telemetry quality.
  • Renamed binaries and custom forks depend more heavily on behavior correlation.
  • Short-lived network bursts can require EDR-grade attribution integration.
  • Evasion remains possible; multi-signal architecture raises cost but does not claim elimination.
  • Class D empirical validation is currently anchored to a single reference implementation.

8) Deployment and Integration

Collector install supports Windows Service, macOS LaunchAgent, and Linux systemd workflows. Configuration is layered: defaults, JSON config, environment variables, then CLI flags. Reporting can run in changes-only mode (default) or report-all mode during rollout and troubleshooting.

No inbound port exposure is required. Agent traffic is outbound HTTPS to the central API.

9) Conclusion

Agentic AI governance requires capability-based detection, confidence-scored decisions, and proportional enforcement. Detec provides this with deterministic scoring, versioned policy logic, lab-backed validation, and explicit publication of known limits.
