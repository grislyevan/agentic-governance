## 5. Agent-Specific Terms

### 5.1 Scanning Behavior

The Agent scans Endpoints for agentic AI tools using five signal layers:

| Layer | What Is Examined | Purpose |
|-------|-----------------|---------|
| Process | Running processes, command lines, process trees | Identify active tool processes |
| File | Installation directories, configuration files, model files | Detect installed tools |
| Network | Active connections, remote endpoints, port usage | Identify tool network activity |
| Identity | OS username, git identity, credential store presence, code signatures | Attribute tool usage |
| Behavior | Activity timestamps, session artifacts, usage patterns | Assess active use vs. stale install |

All five signal layers are evaluated locally on the Endpoint. Raw Telemetry Data (process command lines, file paths, network addresses, git configuration, credential store contents) is used only for local confidence scoring and is **not** included in events transmitted to the Server.

### 5.2 Confidence Scoring

The Agent computes a confidence score between 0.00 and 1.00 for each detected tool using per-tool calibrated weights, penalty adjustments, and evasion-detection boosts. The confidence score and its attribution sources are included in transmitted events. The raw evidence used to compute the score is not transmitted.

### 5.3 Policy Evaluation

The Agent evaluates each detection against the organizational policy to produce an enforcement decision:

- **Detect:** Tool presence is logged; no user-facing action
- **Warn:** Tool presence is logged and the user is notified
- **Approval Required:** Tool use is flagged and requires explicit approval before continued use
- **Block:** Tool use is blocked via configured enforcement mechanisms

Policy rules, thresholds, and enforcement actions are configurable by the Customer.

### 5.4 Endpoint Access Requirements

The Agent requires certain system-level permissions to perform its scanning functions:

- **macOS:** Full Disk Access (for file and process scanning)
- **Windows:** Standard user permissions; elevated permissions for some process scanning
- **Linux:** Standard user permissions; process scanning via `/proc` filesystem

The Agent does not require root or administrator privileges for standard operation, though certain deep-scan capabilities may benefit from elevated access.

### 5.5 Communication

The Agent communicates with the Server using one of two protocols:

- **HTTP (REST API):** JSON payloads over HTTPS to `POST /api/events` and `POST /api/endpoints/heartbeat`, authenticated with an API key
- **TCP (Binary Protocol):** Persistent connection using length-prefixed msgpack frames on a configurable port (default: 8001), authenticated with an API key

When the Server is unreachable, the Agent buffers events locally (default location: `~/.agentic-gov/buffer.ndjson`, maximum 10,000 events) and flushes the buffer when connectivity is restored.

### 5.6 Event Signing

When cryptographic signing is enabled, the Agent signs each event with an Ed25519 private key. The Server can verify event integrity using the corresponding public key registered during endpoint enrollment. Event signing is optional and must be configured by the Customer.

## 6. Server-Specific Terms

### 6.1 Database

The Server supports two database backends:

- **SQLite:** Suitable for single-node and evaluation deployments. Database file stored at a platform-appropriate location (macOS: `~/Library/Application Support/Detec/detec.db`; Linux: `~/.local/share/detec/detec.db`; Windows: `C:\ProgramData\Detec\detec.db`).
- **PostgreSQL:** Recommended for production and multi-node deployments. Connection configured via the `DATABASE_URL` environment variable.

### 6.2 Multi-Tenancy and Data Isolation

The Server enforces tenant-level data isolation. All records (events, endpoints, policies, users, audit logs) are scoped by `tenant_id`. Queries are filtered by tenant at the data access layer. Cross-tenant data access is not possible through the API.

### 6.3 Binary Protocol Gateway

The Server optionally operates a binary protocol gateway for persistent Agent connections. The gateway:
- Listens on a configurable host and port (default: `0.0.0.0:8001`)
- Uses length-prefixed msgpack framing for all messages
- Supports message types: AUTH, EVENT, EVENT_BATCH, HEARTBEAT, POLICY_PUSH, COMMAND, ACK/NACK, ERROR
- Must be explicitly enabled via configuration (`GATEWAY_ENABLED=true`)

### 6.4 Authentication and Authorization

The Server implements:
- JWT-based authentication for Dashboard users with configurable token expiration
- API key authentication for Agent-to-Server communication
- Role-based access control with four roles: Owner, Admin, Analyst, Viewer
- Audit logging of security-relevant actions including actor, action, resource, and client IP

### 6.5 Administration

The Customer is responsible for:
- Initial Server configuration, including setting a strong `JWT_SECRET` and `SEED_ADMIN_PASSWORD`
- User account management and role assignments
- Policy configuration and enforcement rules
- Monitoring Server health and performance
- Database backups and maintenance
